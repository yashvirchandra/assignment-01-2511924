# Part 1 — Normalization Analysis

## Q1.1 Anomaly Analysis

### Insert Anomaly
Suppose the company hires a fourth sales representative who hasn't done any deals yet. In orders_flat.csv, there is no way to store the new person's details because every row requires an order to exist. The sales rep's ID, name, email, and office address are all tied to order rows — so a rep with no or zero sales simply has no place in the table. We would have to insert a fake order just to record the new persosn's details, which leads to corrupts the data. Affected columns: sales_rep_id, sales_rep_name, sales_rep_email, office_address.

### Update Anomaly
Customer C008 (Kavya Rao, kavya@gmail.com) appears across 28 times on different order rows in the file, including ORD1131, ORD1025, ORD1021, ORD1180, and ORD1047 among many others. Her email address kavya@gmail.com is written 28 times, once for every order she have placed. If Kavya ask to update her email address, a developer must correctly update all 28 rows at the exact same time. If even one row gets missed, the database now has two different email addresses for the same customer with no way to tell which one is current email address. This kind of mistake is very hard to detect and can cause real problems like sending confirmations to the wrong email or failed account logins. Affected columns: customer_name, customer_email, customer_city Affected rows: ORD1131, ORD1025, ORD1021, ORD1180, ORD1047 and more.

### Delete Anomaly
Product P008 (Webcam, priced at ₹2100) appears in only one order in the entire dataset: ORD1185. If that order gets deleted for any reason either it was cancelled, it was archived, or the records were cleaned up, every piece of information about the Webcam product is permanently lost. The product name, category, and price all vanish from the database simply because a single order was removed, even though we had no intention of deleting the product itself. Affected columns: product_id, product_name, category, unit_price Affected row: ORD1185.

### Normalization Justification
At first, the manager's argument would make some sense to me that one table, one place to look, no JOINs needed is simpler way. But after actually working on orders_flat.csv, I can see exactly why this approach of the manager doesn't go well.
The file has 186 rows across 8 customers. Customer Kavya Rao appears in more than 28 times in data. Every single row stores her full name, email address, and city — the same information copied over and over. Now imagine this is not 8 customers but 80,000 customers placing orders daily. The storage waste alone is significant, but the real problem is consistency. If Kavya changes her email and the UPDATE statement misses even two rows out of twenty, the database now contradicts itself. Some rows say one email, others say another, and there is no flag or warning telling anyone this happened. Silent data corruption like this is genuinely difficult to detect and fix later.
The delete anomaly is even more alarming. Product P008 (Webcam) exists in just one order. Delete that order and the product is not listed any longer making it unable to found for future entries. There is no product catalog to separate from the orders, the catalog literally lives inside the order rows. This means routine database maintenance could accidentally destroy product information.
Normalizing to 3NF fixes all of this cleanly as each entity gets it's own table customers, products, sales reps, orders and rows in one table simply reference rows in another using IDs. Every piece of information exists in exactly one place. Changing Rohan's email means updating one row in the customers table, and it is immediately consistent everywhere. Yes, queries now require JOINs, but that is a minor inconvenience compared to a database that slowly fills with errors. Normalization is not over-engineering, it is just basic hygiene for any data that is meant to be trusted.
